# Enterprise Design Thinking — Toolkit (facilitation steps)

How to actually run each core activity when [SKILL.md](SKILL.md) is active. For each:
**Purpose** (IBM's intent), **Loop phase**, **Template**, **Prompts to ask the user**,
**Artifact to produce**. Run an activity by explaining it in one line, asking the prompts,
capturing answers, then outputting the filled artifact. Keep fidelity low; iterate.

## Align first

### Hopes and Fears  · Reflect
- **Purpose:** Surface what people hope for and worry about before a project starts.
- **Prompts:** For this work, what do you most hope happens? What do you most fear? (Ask
  per stakeholder/persona if more than one.)
- **Artifact:** Two clustered columns — **Hopes** | **Fears** — with themes grouped and
  the top 1–2 of each flagged to address.

### Assumptions and Questions  · Reflect
- **Purpose:** Reality-check — separate what you *believe* from what you *know*.
- **Template:** A 2×2 grid — vertical axis **High-risk → Low-risk**, horizontal axis
  **Certain → Uncertain**. Plot each assumption/question, one per note.
- **Prompts:** What are we assuming or guessing about? For each: how certain are we of the
  answer, and how risky is it if we're wrong?
- **Artifact:** The plotted grid. Focus on the **high-risk + uncertain** quadrant — those
  are the items to validate next (turn them into Observe/research activities).

### Stakeholder Map  · Reflect
- **Purpose:** Identify stakeholders, their expectations, and their relationships.
- **Steps:** List stakeholders one per note (teams, roles, leads, execs, partners,
  customers, end users). For each, add a second note with a **quote** in their voice
  expressing their thoughts or expectations. Cluster into labelled groups, then draw and
  label relationship lines (influence, process, dependency) among the groups.
- **Prompts:** Who's affected by or invested in this? In their own words, what does each
  one expect? Who influences or depends on whom?
- **Artifact:** Clustered, labelled stakeholder groups, each with a representative quote,
  and relationship lines between groups. (Assume someone is involved until proven otherwise.)

## Observe → Synthesize

### Empathy Map  · Reflect (on Observe data)
- **Purpose:** Build empathy for a user from the team's observations.
- **Template:** Four quadrants around the user — **Says | Thinks | Does | Feels**.
- **Prompts:** From what we observed, what did this user *say*? What were they likely
  *thinking*? What did they *do*? What did they *feel*? Note contradictions (says one thing,
  does another) — those are insights. **Annotate unknowns** (assumptions and questions) to
  validate later.
- **Artifact:** A filled 4-quadrant map per key persona (user sketched in the centre with a
  name), surprises/contradictions called out, and unknowns flagged for inquiry.

### As-is Scenario Map  · Reflect
- **Purpose:** Understand the user's *current* experience, step by step.
- **Template:** Phases across the top; for each phase three rows — **Doing | Thinking |
  Feeling**.
- **Prompts:** What are the phases of how they do this today? In each phase: what are they
  doing, thinking, and feeling? Where are the lowest emotional points?
- **Artifact:** A phase-by-phase grid with an emotional low-point (pain) marked — those
  pains are where ideas should aim.

### Needs Statements  · Reflect
- **Purpose:** Outline what users need to reach their goals (solution-neutral).
- **Template:** *"[A user] needs a way to [need / verb], because/so that [insight or
  goal]."*
- **Prompts:** What does this user fundamentally need to accomplish? Why does it matter to
  them? (State the need, never a feature.)
- **Artifact:** A short list of needs statements, each tied to an observed insight.

## Plan / align direction

### Hills  · Reflect (Key)
- **Purpose:** State intent as a meaningful user outcome — the team's north star.
- **Template:** **Who + What + Wow**, implementation-independent.
  - *Who* — specific user/group. *What* — new enablement. *Wow* — measurable,
    differentiating edge.
- **Prompts:** Who exactly? What will they be able to do that they can't today? What's the
  wow that makes it remarkable and how would we measure it? Does this name an outcome, not
  a solution? (If it names a feature, rewrite it.)
- **Artifact:** **Three (and only three) Hills** — IBM's explicit guidance. Each is one
  sentence. Valid extra ideas go onto a roadmap for a future release, not a fourth Hill.
  Example shape: *"A [who] can [what] [wow]."*

### Prioritization Grid  · Reflect
- **Purpose:** Decide the next move at the intersection of importance and feasibility.
- **Template:** 2×2 — X axis **Feasibility** (for the team), Y axis **Importance** (to the
  user). Plot ideas/needs.
- **Prompts:** For each candidate: how important to the user (low→high)? how feasible for
  us (low→high)?
- **Artifact:** A populated grid using IBM's zones — high importance + high feasibility =
  **no-brainers** (do now); high importance + low feasibility = **big bets** (make more
  feasible); low importance + high feasibility = **utilities** (make more impactful); low
  importance + low feasibility = **unwise** (skip). Competitors also chase the no-brainers,
  so differentiate on the big bets.

## Make / ideate

### Big Idea Vignettes  · Make
- **Purpose:** Rapidly diverge on a breadth of possible solutions — quantity over polish.
- **Template:** Each *vignette* is a **pair** — one note with a 1–2 word headline, a second
  note with a quick visual sketch (a single storyboard frame).
- **Prompts:** Given this Hill/need, generate many distinct directions. Push past the first
  three obvious ones (wild + safe + adjacent). Don't evaluate while generating ("say yes to
  the mess").
- **Artifact:** Many headline+sketch vignettes, then clustered. Take the strongest into
  Scenario Maps or Storyboarding — but score them on a Prioritization Grid first.

### To-be Scenario Map  · Make
- **Purpose:** Vision of the user's *improved future* experience with your idea in it.
- **Template:** Same structure as As-is (Phases × Doing/Thinking/Feeling), projected to the
  ideal future.
- **Prompts:** With our idea in place, what are the new phases? What do they now do, think,
  feel? How do the old as-is pain points resolve?
- **Artifact:** A future-state grid, ideally diffed against the As-is map to show the
  improvement at each pain point.

### Storyboards  · Make
- **Purpose:** Communicate an idea as a visual story showing how it fits the user's life.
- **Prompts:** Walk the idea as 4–8 frames: the trigger, the user's steps, the moment of
  value (the wow), the outcome. Who, where, what changes?
- **Artifact:** A numbered frame-by-frame outline (caption + what's shown per frame) ready
  to sketch.

## Test

### Feedback Grid  · Make / Reflect
- **Purpose:** Gather and organize feedback into action after a Playback or test.
- **Template:** 2×2 — **What worked | What to change | Questions | Ideas (new)**.
- **Prompts:** From this feedback/test: what worked? what should change? what open
  questions surfaced? what new ideas appeared?
- **Artifact:** A filled 4-quadrant grid; convert "change" + "questions" into next-loop
  actions.

## Continuous alignment

### Playbacks  · Reflect (Key)
- **Purpose:** Tell the story so far, share work, exchange feedback, expose misalignment.
- **How to run:** Narrate as a story, not a status report: **user → their need → our idea →
  the outcome against the Hill.** Then explicitly invite critique and capture it (use a
  Feedback Grid).
- **Artifact:** A short story-form playback script + captured feedback + decisions.

### Retrospectives  · Reflect
- **Purpose:** At the end of an iteration, improve *how the team works*, not the product.
- **Prompts:** What should we keep? stop? start? What slowed us down?
- **Artifact:** Keep / Stop / Start list with 1–2 committed changes for the next iteration.

## Simulated runs (subagents): Sponsor Users + saturation

Use when the runtime has an Agent tool and you want to rehearse the Loop on any topic
*before* (never *instead of*) reaching real users. See the matching procedure in
[SKILL.md](SKILL.md).

### Simulated Sponsor-User Observe round  · Observe
- **Honesty rule:** simulated users are NOT real; their output is a set of assumptions to
  validate, never validation. State this in every playback.
- **Cast:** ~5 personas within ONE target segment, spanning its range (confidence,
  attitude, context, motivation). Brief each as a user being interviewed — honest,
  objections welcome, not there to please you. Spawn in parallel.
- **Probe:** the riskiest assumption + the loudest fears, via a consistent question set so
  answers compare cleanly.
- **Saturation check:** track the **new themes each user adds**; stop when new themes → ~0
  (usually by user 4–5). Add 1–2 more personas if still diverging. Show the taper. (The
  same saturation logic applies to real interview rounds.)
- **Output:** converged pattern (≥4 of 5) → scorecard vs. assumptions → needs statements →
  proceed to Make. Always close by naming the real-user test.

## Pointers (other toolkit activities, used as needed)
- **Contextual Inquiry / Interviews** (Observe) — first-hand user data; the source of
  truth that feeds Empathy Maps and As-is maps.
- **Experience-based Roadmap / Research Plan / User Stories** (Plan) — sequence outcomes,
  plan research, express requirements as user tasks.
- **Speculative Design Stories** (Make) — provoke with plausible future problems.
- **Paper Prototypes / Wireframes / Cognitive Walkthrough** (Test) — make ideas tangible
  at low fidelity and evaluate them with users.
- **Daily Syncs** (Align) — surface blockers early.
- **AI Essentials: Intent / Data / Understanding / Reasoning / Knowledge** — run when the
  candidate solution may involve AI, starting with *Intent* (should AI solve this at all?).
